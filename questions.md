# Вопросы для консультации с преподавателем

## 1. Архитектура хранилища

При переходе с `NonEmptyVec<T>` на `HashMap<String, T>` теряется гарантия непустой коллекции
(сейчас `NonEmptyVec` на уровне типов запрещает пустой дом или пустую комнату). Стоит ли:

- Полностью убрать гарантию — пустой `HashMap` допустим?
- Или оставить проверку в конструкторе, но уже на уровне рантайма (возвращать `Result` при создании пустой комнаты)?

---

## 2. Судьба `RoomId(usize)` и `DeviceId(usize)`

После перехода на строковые ключи ньютайп-обёртки `RoomId` и `DeviceId` теряют смысл.
Их нужно полностью удалить, или они могут сохраниться для других целей (например, порядкового перебора)?

---

## 3. Ошибка в методе получения устройства по имени

Задание говорит, что метод `get_device(room_name, device_name)` должен возвращать тип ошибки,
реализующий `std::error::Error`. В проекте уже есть `InputError` через `thiserror`. Правильно ли:

- Добавить новые варианты (`RoomNotFound(String)`, `DeviceNotFound(String)`) в существующий `InputError`?
- Или создать отдельный тип ошибки специально для этого метода?
- Допустимо ли продолжать использовать `thiserror`, или предполагается ручная реализация трейта `std::error::Error`?

---

## 4. Сигнатура трейта для отчёта

Задание требует вынести отчёт в трейт. Какой предпочтительный вариант сигнатуры:

```rust
trait Report {
    fn report(&self) -> String;  // возвращает строку
}
```

или

```rust
trait Report {
    fn print_report(&self);  // печатает напрямую
}
```

И что предпочтительнее для функции в `app`, принимающей «что-то умеющее выводить отчёт» —
`impl Report` (статическая диспетчеризация) или `&dyn Report` (динамическая)?

---

## 5. Макрос создания комнаты

Задание описывает макрос, принимающий пары `(ключ, объект_розетки)` или `(ключ, объект_термометра)`.
Несколько уточнений:

- Должен ли макрос принимать именно `SmartSocket`/`SmartThermometer` и преобразовывать их через
  `From` в `SmartDevice` внутри себя?
- Или макрос работает уже с готовыми `SmartDevice`?
- Ожидаемый синтаксис вызова — что-то вроде
  `room!("socket1" => SmartSocket::default(), "thermo1" => SmartThermometer::default())`?
- Где должен быть определён макрос — в `home` (и экспортирован) или в `app`?

---

## 6. `Debug` — дериватив или ручная реализация

Для большинства типов `#[derive(Debug)]` достаточно. Есть ли случаи, где ожидается ручная
реализация `fmt::Debug` (например, для красивого вывода `SmartDevice` или скрытия внутренних полей)?

---

## 7. Добавление и удаление — возврат ошибки

При добавлении устройства/комнаты с уже существующим ключом что предпочтительнее:

- Тихая перезапись (как `HashMap::insert`)?
- Возврат `Err(...)` с сообщением о дублировании?

А при удалении несуществующего ключа — возвращать `Option` или `Result`?

---

## 8. Обратная совместимость API

Текущий публичный API использует числовые индексы (`get_room(RoomId)`, `get_device(DeviceId)`).
После рефакторинга API полностью меняется на строковые ключи. Нужно ли сохранять старый API
в каком-либо виде, или предполагается полная замена?

---

## 9. Тесты

Задание требует привести тесты в соответствие с новым функционалом. Ожидается ли:

- Только обновление существующих тестов под новые сигнатуры?
- Или добавление новых тестов на add/remove, макрос, трейт отчёта, обработку ошибок?
- Нужно ли тестировать `From`-конверсии?

---

## 10. Фича-флаги `mock` / `real`

Задание не затрагивает фича-флаги. После рефакторинга они остаются?
Или предполагается упростить/убрать разделение на `mock.rs` и `real.rs`?

---

## 11. `anyhow` в `app`

Сейчас `app` использует `anyhow` для обработки ошибок. После того как `home` будет иметь
собственный тип ошибки, реализующий `std::error::Error`, — уместно ли продолжать использовать
`anyhow` в бинарнике, или предпочтительнее работать с конкретными типами ошибок?
